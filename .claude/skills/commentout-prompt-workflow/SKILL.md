# Skill: @prompt ディレクティブ駆動開発

## 概要
ソースコード内に埋め込まれた `@prompt` ディレクティブを起点として開発を行うワークフロー。
コードがブラックボックス化することを防ぐための仕組み。

## トリガー
ユーザーが `/prompt` を実行した時、または @prompt ディレクティブ駆動の指示があった時

## プロンプト記法

開発者はソースコード内のコメントブロックに `@prompt` を記述して指示を埋め込む。

```
/**
 * @brief 関数の概要説明
 * @prompt ここにClaude Codeへの指示を自然言語で記述する。
 * 複数行にわたって記述可能。
 */
```

## フロー

### Phase 1: 差分検出と @prompt 集計

1. `git diff HEAD` で差分のあるファイルを確認する
   - ステージング済み・未ステージングの両方を確認する場合は `git diff` と `git diff --cached` を併用する
   - 差分が存在しない場合は `git diff HEAD~1` で直前コミットとの差分を確認する
2. 差分のあるファイルを対象に、すべての `@prompt` ディレクティブを集計する
   - 変更のないファイルに含まれる `@prompt` は**対象外**とする（差分駆動）
   - 同一ファイル内に複数の `@prompt` がある場合は出現順にリストする
   - 依存関係がある場合（例: 関数Aの結果を関数Bが使う）は依存順を注記する

### Phase 2: 計画書作成とレビュー

3. 集計結果をもとに `.claude/plans/` 配下に計画書を作成する
4. 計画書内には以下の形式で集計したプロンプト一覧を**必ず残す**:

```markdown
## 対象プロンプト一覧

| # | ファイル | 対象 | 指示概要 |
|---|---------|------|---------|
| 1 | src/example.ts:12 | test() | hello worldを出力する関数を作成 |
```

5. plan作成後、自動で `codex` スキルを使用してplanレビューを依頼する（ユーザーの指示を待たない）
6. Codex が `GO` を出すまで、**plan.md のみ**を修正する
7. `GO` が出るまで **実装してはいけない**

### Phase 3: 実装

8. 計画書の順序に従い、1つずつ完了させる
9. 実装完了後、対応する `@prompt` コメントをソースコードから削除する
10. 計画書のステータスを随時更新する
11. 実装後、必ずテストを実行する（**Skill**を使用）

### Phase 4: 完了

12. 変更をステージする
13. コミットする前に承認を求める
14. 変更をコミットする（git-rule.md のコミットメッセージ規約に従う）
