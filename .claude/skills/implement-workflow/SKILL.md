# Skill: コーディングワークフロー

## 概要
実装タスクを Plan → Implementation → Completion の3フェーズで進めるワークフロー。

## トリガー
ユーザーから実装を伴うタスクが指示された時（自動適用）

## フロー

### Phase 1: Plan Loop（設計合意）

1. 指示されたタスクの内容を理解する
2. 実装を必要としないタスク（コードの理解、レビュー等）の場合はそのまま実行して終了する
3. 実装が必要な場合:
   - `.claude/plans/` 配下に `plan-<UTCに基づいた識別子>.md` を作成する
   - plan作成後、自動で `codex` スキルを使用してplanレビューを依頼する（ユーザーの指示を待たない）
   - Codex が `GO` を出すまで、**plan.md のみ**を修正する
   - `GO` が出るまで **実装してはいけない**

### Phase 2: Implementation Loop（実装）

4. Codex が `GO` を出したら、実装を開始する
5. 以下の品質観点を守る:
   - 小さな変更単位で単一責務を保つ
   - 影響範囲を最小化する
   - コードの可読性・保守性を確保する
   - セキュリティリスクを排除する
   - パフォーマンスへの影響を考慮する
   - テストカバレッジを維持する
   - 必要に応じてドキュメントを更新する
   - 既存の設計パターン・コーディングスタイルに従い、新規作成前に既存実装を確認する
   - 複雑なロジックには日本語コメントを追加する
   - 適切な例外処理を実装する
   - 機密情報（APIキー、パスワード等）はハードコードせず環境変数を使用する
6. 実装後、必ずテストを実行する（**Skill**を使用）

### Phase 3: Completion（完了）

7. 変更をステージする
8. コミットする前に承認を求める
   - ユーザーから追加の指示がある場合は、その内容に従い必要であれば Phase 1 から再度繰り返す
9. 変更をコミットする（git-rule.md のコミットメッセージ規約に従う）
