## テスト戦略

### ユニットテスト
- **対象**: ビジネスロジック、ユーティリティ関数、カスタムフック
- **ツール例**: Jest, Vitest, React Testing Library
- **方針**:
  - 純粋関数は入出力を網羅的にテストする
  - 副作用のあるロジックはモックを活用する
  - カバレッジ目標: 主要なビジネスロジックは80%以上を目指す

### 統合テスト
- **対象**: API エンドポイント、モジュール間の連携
- **ツール例**: Supertest (NestJS), MSW (フロントエンド)
- **方針**:
  - 正常系・異常系の主要パスをカバーする
  - 外部サービス（DB, 認証等）はモックまたはテスト用インスタンスを使用する

### E2Eテスト
- **対象**: ユーザーの主要操作フロー
- **ツール例**: Playwright, Cypress
- **方針**:
  - クリティカルパス（ログイン、主要機能の一連の操作）を中心にテストする
  - 必要十分な量に留め、メンテナンスコストとのバランスを取る

### テスト実行ルール
- プルリクエスト作成時にCIで自動実行する
- テストが失敗している状態でのマージは禁止する
- 新機能追加時は対応するテストも併せて作成する
