## エラーハンドリング・ログ方針

### エラー分類

| 分類 | 説明 | 対応方針 |
|---|---|---|
| ユーザーエラー | バリデーション失敗、不正な操作等 | ユーザーにわかりやすいメッセージを返す。ログレベル: `warn` |
| システムエラー | アプリケーション内部の予期しない障害 | 内部詳細を隠蔽し、汎用メッセージを返す。ログレベル: `error` |
| 外部サービスエラー | DB接続失敗、外部API障害等 | リトライ可否を判断し、適切にフォールバックする。ログレベル: `error` |

### ログレベル基準

| レベル | 用途 | 例 |
|---|---|---|
| `debug` | 開発時のみ必要な詳細情報 | 変数の中間状態、SQL発行内容 |
| `info` | 正常動作の記録 | リクエスト受信、処理完了、ユーザーログイン |
| `warn` | 想定内だが注意が必要な事象 | バリデーション失敗、非推奨APIの使用 |
| `error` | 即座に対応が必要な障害 | 未捕捉例外、外部サービス接続失敗 |

### ログフォーマット

- 構造化ログ（JSON形式）を使用する
- 以下のフィールドを必ず含める:
  - `timestamp` — ISO 8601 形式
  - `level` — ログレベル
  - `message` — 人間が読めるメッセージ
  - `context` — 発生元のモジュール・クラス名
  - `traceId` — リクエスト単位のトレースID（リクエストの追跡用）

```json
{
  "timestamp": "2025-01-15T10:30:00.000Z",
  "level": "error",
  "message": "Failed to connect to Firestore",
  "context": "DeviceService",
  "traceId": "abc-123-def"
}
```

### エラーレスポンスの統一フォーマット

- APIから返すエラーレスポンスは以下の形式に統一する:

```json
{
  "statusCode": 400,
  "error": "Bad Request",
  "message": "デバイスIDは必須です",
  "details": []
}
```

- `details` にはバリデーションエラー等の補足情報を配列で格納する
- スタックトレースや内部実装の詳細はレスポンスに含めない

### 実装ルール

- **フロントエンド**: グローバルエラーバウンダリ（`error.tsx`）で未捕捉エラーを処理する
- **バックエンド**: NestJSのExceptionFilterでエラーレスポンスを統一する
- **try-catchの乱用を避ける**: エラーが発生しうる箇所のみで捕捉し、握りつぶさない
- **エラーの再スロー**: 捕捉後にログ記録だけして再スローする場合、元のエラー情報を保持する
- **ユーザー向けメッセージ**: 技術的な詳細ではなく、次のアクションを示すメッセージにする

### 監視・アラート

- （例: Sentry, Datadog, CloudWatch 等の監視ツールを記載する）
- `error` レベルのログはアラート通知の対象とする
- アラートの通知先と対応フローを別途定める
