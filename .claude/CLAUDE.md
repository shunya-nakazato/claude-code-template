## 9. タスク実行方針

### エージェント基本方針
1. **サブエージェント**: コンテキストまたは処理負荷の高いタスクはサブエージェントを使用する
2. **即時実行**: CLAUDE.md にワークフローが定義されている場合、コードベースの広範な探索を行わず即座にワークフローに従って実行する。探索は必要最小限に留め、計画フェーズに過度な時間をかけない

### 開発時の基本方針
1. **日本語使用**: コメント、ログ、ドキュメントはすべて日本語
2. **型安全**: 型を厳格に定義
3. **テスト駆動**: 機能実装前にテストケースを検討
4. **段階的実装**: 大きな機能は小さなステップに分割
5. **変更前確認**: 既存コードへの影響を事前評価

### コード生成・修正時のルール
- **既存コードの尊重**: 既存の設計パターンやコーディングスタイルに従う
- **既存コード優先**: 新しいユーティリティ関数・定数・ヘルパーを作成する前に、既存の実装を必ず確認する。隣接ファイルのインポートパスを参考にし、プロジェクト内の定数定義（`constants/`配下）を優先的に使用する
- **バックアップ不要**: Gitで管理されているため、直接編集可
- **コメント追加**: 複雑なロジックには日本語コメントを追加
- **エラーハンドリング**: 適切な例外処理を実装

### 禁止事項
- **機密情報のハードコード**: APIキー、パスワード等は環境変数を使用

## 10. @prompt ディレクティブ駆動開発

ソースコード内に埋め込まれた `@prompt` ディレクティブを起点として開発を行う仕組み。
コードがブラックボックス化することを防ぐため、このワークフローを**必ず順守**すること。

### プロンプト記法

開発者はソースコード内のコメントブロックに `@prompt` を記述して指示を埋め込む。

```
/**
 * @brief 関数の概要説明
 * @prompt ここにClaude Codeへの指示を自然言語で記述する。
 * 複数行にわたって記述可能。
 */
```

### 記法ルール

- `@prompt` はブロックコメント (`/** ... */`, `/* ... */`, `<!-- -->`, `# ---` 等) 内に記述する。
- `@prompt` の直後から、コメントブロック終端または次のタグ (`@brief`, `@param` 等) までを指示本文とみなす。
- 1ファイルに複数の `@prompt` が存在してもよい。
- `@prompt` が付与された関数・クラス・ブロックの**既存コード**は実装対象のスケルトンまたはコンテキストとして扱う。

### @prompt 駆動時の計画書内容

計画書内には以下のような形で集計したプロンプトを**必ず残すこと**。

```markdown
## 対象プロンプト一覧

| # | ファイル | 対象 | 指示概要 |
|---|---------|------|---------|
| 1 | src/example.ts:12 | test() | hello worldを出力する関数を作成 |
```

## 11. タスク実行手順 (イベント駆動)

以下のイベント駆動フローに厳密に従ってください。

---

### Phase 1: Plan Loop（設計合意）
1. あなたは指示されたタスクに基づき、その内容を理解して次のいずれかを実行する
    a. 実装が必要なタスクの場合（通常）
        a-1. .claude/plans配下に `plan-<UTCに基づいた識別子>.md` を作成する
        a-2. plan作成後、自動で `codex` スキルを使用してplanレビューを依頼する（ユーザーの指示を待たない）
        a-3. Codex が `GO` を出すまで、**plan.md のみ**を修正する
        a-4. `GO` が出るまで **実装してはいけない**
        a-5. Phase 2 に進む
    b. 実装を必要としないタスクの場合 (例：ソースコードの理解、レビュー等)
        b-1. タスクを実行して終了
    c. @prompt ディレクティブ駆動のタスクの場合
        c-1. `git diff HEAD` で差分のあるファイルを確認する
             - ステージング済み・未ステージングの両方を確認する場合は `git diff` と `git diff --cached` を併用する
             - 差分が存在しない場合は `git diff HEAD~1` で直前コミットとの差分を確認する
        c-2. 差分のあるファイルを対象に、すべての `@prompt` ディレクティブを以下の形式で集計する
             ```
             [ファイルパス]:[行番号]
               対象: <関数名 / クラス名 / ブロック名>
               指示: <@prompt の本文>
             ```
             - 変更のないファイルに含まれる `@prompt` は**対象外**とする（差分駆動）
             - 同一ファイル内に複数の `@prompt` がある場合は出現順にリストする
             - 依存関係がある場合（例: 関数Aの結果を関数Bが使う）は依存順を注記する
        c-3. 集計結果をもとに .claude/plans 配下に計画書を作成する（セクション10のテンプレートを使用）
        c-4. plan作成後、自動で `codex` スキルを使用してplanレビューを依頼する（ユーザーの指示を待たない）
        c-5. Codex が `GO` を出すまで、**plan.md のみ**を修正する
        c-6. `GO` が出るまで **実装してはいけない**
        c-7. Phase 2 に進む

---

### Phase 2: Implementation Loop（実装）
5. Codex が `GO` を出したら、実装を開始する
6. 実装は以下のルールを守る
    - 小さな変更単位
    - 単一責務
    - 影響範囲を最小化
    - コードの可読性・保守性
    - セキュリティリスクの有無
    - パフォーマンスへの影響
    - テストカバレッジ
    - ドキュメント更新の必要性
7. @prompt 駆動の場合は追加で以下を守る
    - 計画書の順序に従い、1つずつ完了させる
    - 実装完了後、対応する `@prompt` コメントをソースコードから削除する
    - 計画書のステータスを随時更新する
8. 実装後、必ずテストを実行する（**Dockerコンテナ上で実行**）
    - テストコマンド: `./.claude/scripts/test.sh [target]`
    - target: `all` | `frontend` | `backend` | `lint` | `type-check`
    - 例: `./.claude/scripts/test.sh all` （全テスト実行）

---

### Phase 3: Completion（完了）
9. 変更をステージする
10. コミットする前に承認を求めてください. 
    - ユーザーから追加の指示がある場合は、その内容に従い必要であれば Phase 1 から再度繰り返してください
11. 変更をコミットする（Conventional Commits形式）
    - **フォーマット**:
      ```
      <type>(<scope>): <title>

      <body>

      Ref: plan-<識別子>.md
      ```
    - **type一覧**:
      - `feat:` 新機能
      - `fix:` バグ修正
      - `docs:` ドキュメント
      - `style:` フォーマット
      - `refactor:` リファクタリング
      - `test:` テスト
      - `chore:` ビルド、設定変更
    - **禁止事項**:
      - 余計なフッター（`Generated with...`, `Co-Authored-By`等）は含めない
      - 絵文字は使用しない
